# 概念卡提炼 + 章节切分 + wikilink 修复指南

> 配套 `book-deep-analysis` SKILL.md Step 3-6 实战细节

## Part 1 · 章节切分

OCR 出的全本是连续 Markdown，带 `<!-- page=N -->` 标记。按 `第N章` 标题切分。

### 切分正则
```python
import re
# 兼容空格："第 1 章" / "第1章" / "第 1 章"
pattern = re.compile(r'^#+\s*第\s*(\d+)\s*章', re.MULTILINE)
```

### 切分逻辑
1. 用正则找到所有 `第N章` 的行号
2. 从一个章节起始行到下一个章节起始行 = 一章
3. 文件名格式：`第01章_章名.md`（2 位数补零）

### 常见坑
- **目录页 OCR 出的章节标题 ≠ 正文章节标题** —— 以正文标记为准
- **章节标题有全角/半角空格** —— 正则加 `\s*` 兼容
- **正则只匹配 `^#+\s*第N章`** —— 如果章节标题没有 `#` 前缀，OCR prompt 要强调标题格式

## Part 2 · 概念卡提炼（GLM-5.2）

### 为什么要用纯文本模型
- OCR 出来的是文本 → 概念提炼是文本任务 → 用纯文本模型（GLM-5.2）
- **不要用视觉模型（GLM-4.6V）做概念提炼** —— 视觉模型提炼质量不如纯文本模型
- 两件事分开干，效率最高

### GLM-5.2 输出 JSON 解析
GLM-5.2 输出的 JSON 经常有以下问题：
1. **包裹在 ```json 代码块里** —— 先提取再解析
2. **JSON 截断**（max_tokens 不够） —— 找最后一个 `},` 补 `]`
3. **嵌套引号** —— `json.loads` 会报错，需要宽松解析

宽松解析策略见 `scripts/extract_concepts.py` 的 `parse_concepts()` 函数。

### 概念卡覆盖率
- GLM-5.2 自动生成 ~70%
- **漏的 30% 恰好是全书最核心的概念**
- 必须自己读一遍补上，否则白拆了

## Part 3 · wikilink 4 步修复

Obsidian wikilink 有 4 个常见问题，用 `scripts/fix_wikilinks.py` 一键修复：

| 问题 | 表现 | 修复 |
|---|---|---|
| 冒号不匹配 | wikilink 写半角 `:`，文件名用全角 `：` | wikilink 里的冒号改全角 |
| 相对路径 | `[[../章节/xxx]]` 无法解析 | 改成完整路径 |
| 死链占位符 | `[[02_章节目录]]` 不存在 | 改成 `[[0_总览]]` |
| 第0章 fallback | `[[第0章]]` 不存在 | 改成 `[[0_总览]]` |

## Part 4 · 长路径 → 短路径

Obsidian Graph View 解析短路径 wikilink 更稳定。

### 对比
```
完整路径: [[30_书籍阅读/书名/概念/第12章_三篮分类法]]
短路径:   [[第12章_三篮分类法]]
```

### 效果
- 长路径版本：646 条卡→卡连接（Obsidian 偶发解析失败）
- 短路径版本：728 条连接（+13%）

用 `scripts/shorten_wikilinks.py` 一键转换。
